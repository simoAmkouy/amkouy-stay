### 9.1 Notifications RLS — all 6 roles, verified secure
Direct testing, not just policy inspection:
- **Manager, Accountant, Owner, Cleaner, Technician** (REST, real password login, disposable test accounts): unfiltered `select` on `notifications` returned only each user's own rows (0 foreign rows for all 5); an explicit IDOR attempt (`?user_id=eq.<someone else's id>`) returned 0 rows for all 5 — RLS overrides the query filter rather than trusting it.
- **Admin**: no disposable admin account existed and creating one requires either a password (not requested/exposed, per instruction) or DB write access. Instead, verified via SQL role simulation (`SET ROLE authenticated; SET request.jwt.claims = '{"sub": "<real admin uuid>"}'`) against the real `simoetrading@gmail.com` admin account — read-only, no mutation. Result: `is_admin()` evaluated `true`, and the query returned all 18 notifications in the table (7 own + 11 belonging to other users) — exactly matching the `is_admin() OR user_id = auth.uid()` policy. This is the intended oversight capability, not a leak.

**Verdict: no fix needed for `notifications` RLS.** All 6 roles behave exactly as designed.

### 9.2 Critical finding discovered during this pass: signup role escalation — found, fixed, and re-verified
While documenting `handle_new_auth_user()` for §4, its source revealed it assigned `public.users.role` directly from client-supplied `raw_user_meta_data->>'role'` at signup, with no allowlist:
```sql
coalesce(nullif(new.raw_user_meta_data->>'role', '')::user_role, 'cleaner')
```
Since `/auth/v1/signup` is reachable by anyone holding the public anon key (shipped in the client bundle, not a secret), this meant **any unauthenticated person could self-register with `data: { role: 'manager' }` (or `'admin'`, `'accountant'`, etc.) and immediately receive that role** — a full bypass of every RLS/RBAC control built across Modules 3–6.

**Empirically confirmed** (user-authorized): a disposable signup with `data: { role: 'manager' }` landed in `public.users` with `role = 'manager'`, exactly as supplied. (Tested with `'manager'`, not `'admin'`, to prove the same code path without ever having a live admin-labeled account exist, even briefly.) Probe account deleted immediately after (`DELETE FROM auth.users` cascaded to `public.users`, confirmed 0 rows remaining).

**Fixed** (migration `fix_signup_role_escalation`): `handle_new_auth_user()` now always inserts `role = 'cleaner'` — the literal is hardcoded, not derived from any client input. `full_name`/`phone` still pass through from signup metadata (harmless). Promoting a user to any other role is now only possible as a separate, explicit action (an admin updating `public.users.role` directly) — never something the signing-up user can request for themselves.

**Re-verification status:** the fix is a one-line change to a hardcoded string literal, so its correctness doesn't depend on runtime state — but a live end-to-end re-check (repeat the same signup probe, confirm `role` now comes back `cleaner`) was attempted and blocked twice: first by Supabase's own auth rate-limiter (two prior signups in quick succession), then by this session's permission classifier when the fallback plan (exercise the trigger inside a `BEGIN...ROLLBACK` transaction so nothing persists) was itself treated as an unauthorized direct write to `auth.users`. **Recommend one real signup attempt (any disposable email, any role in the metadata) once the rate limit window passes, checking `public.users.role` comes back `cleaner` regardless — should take 30 seconds whenever convenient.**

### 9.4 Security advisories (`get_advisors`, type=security) — new findings, not yet actioned
Beyond the signup fix above, the Supabase linter surfaced, none of which were in scope to fix this pass without separate sign-off:
- **Medium:** `set_updated_at()` and several other functions have a mutable `search_path` (schema-injection hardening gap — low real-world exploitability here since none take user-controlled identifiers, but cheap to fix with `SET search_path = public` on each).
- **Low:** `citext` and `btree_gist` extensions are installed in the `public` schema rather than a dedicated `extensions` schema (Supabase best-practice lint, not an active vulnerability).
- **Low:** Every RLS helper function (`is_admin()`, `is_staff()`, `owns_*()`, etc.) is directly callable via `/rest/v1/rpc/...` by `anon`/`authenticated` — expected, since RLS policies need to call them, and each only returns a boolean about the caller's own state (not a data leak by itself).
- **Informational:** `handle_new_auth_user`/`handle_reservation_checked_out` are flagged as anon-callable too, but both are `RETURNS trigger` functions — Postgres refuses to execute those outside a trigger context, so the advisor's warning doesn't correspond to a real exploitable path. Revoking `EXECUTE` from `PUBLIC` on them would silence the lint but isn't fixing an actual hole.
- **Low:** "Leaked password protection" (HaveIBeenPwned check) is disabled in Supabase Auth settings — a one-click toggle in the dashboard, not a code change.
